# CTF WP Writer

## 目标

把“解题过程”整理成一份能复现的中文 WP，同时交付一份不依赖命令行参数的 `Python` EXP 脚本；产物统一落在当前题目的 `work/` 目录。

## 交付物约定（强约束）

- 只在当前题目目录下创建/使用 `work/`
- WP：`work/<题目名>_wp.md`
- EXP：`work/exp.py`（如需多脚本，最终只保留一个可一键运行的主脚本，其它临时文件在结束前删除）

## 写作风格

- 中文、像人写的：按“现象 -> 判断 -> 验证 -> 结论”推进
- 少用口号/模板句；多写当时看到的具体响应、关键字段、为什么这么猜
- 结论只基于证据（抓包、源码、报错栈、行为差异）

## 红线与脱敏

- WP 中不写靶机域名/IP/端口：统一留空（必要时用空字符串变量或占位描述，但不出现真实地址）
- flag 只在对话里单独输出给用户；不要写进 WP

## 通用工作流（推荐按顺序执行）

### 1) 建立题目工作区

- 在题目目录创建 `work/`
- 约定题目名（用于 WP 文件名），并记录开始时间（用于最终耗时输出）

### 2) 快速确定题型与攻击面

根据题目附件/源码/流量/二进制等，先回答三个问题：

- **入口**：从哪里输入可控数据（参数、JSON、Header、Cookie、文件、模板变量等）
- **边界**：哪些校验/鉴权/过滤会挡住利用（登录态、WAF、长度限制、编码、黑名单）
- **目标**：flag 位于哪里（文件、数据库、接口返回、容器环境变量、命令执行回显等）

输出到 WP 的“推理过程”时，要把“为什么怀疑这里”写清楚（比如：错误回显、拼接痕迹、可控字段进入危险函数）。

### 3) 漏洞定位（必须落到代码链路/请求链路）

WP 里至少要有：

- 关键路由/函数/类的名字
- 关键参数从请求到危险点的流向（最好用 3-6 行的伪代码/摘录，别整段粘贴）
- 利用成立的条件（比如：必须先注册、必须是管理员、必须绕过某字段校验等）

### 4) 构造利用（先手动可复现，再脚本化）

- 先用最小请求复现漏洞（能稳定触发/回显/side effect）
- 再把它固化为 EXP：
  - 不接收命令行参数
  - 目标地址用空字符串常量（例如 `BASE_URL = ""`）
  - 必要的账号口令、token、路径等也写在脚本顶部，默认留空或可直接注册
  - 运行后自动完成：注册/登录 -> 构造 payload -> 触发 -> 提取 flag -> `print(flag)`

### 5) 写 WP（边写边校验“能复现”）

WP 建议结构（可以按题型微调）：

1. 题目概览（题型、目标、附件）
2. 信息收集与入口确认（关键请求/响应）
3. 漏洞定位与成因（代码链路/数据流）
4. 利用思路与关键 payload（为什么这样构造）
5. 复现步骤（从注册/登录到拿到 flag 的完整操作）
6. 一键 EXP（完整代码）
7. 小结（绕过点/坑/稳定性处理）

### 6) 收尾检查与清理

- WP 内全文检索：不应出现 IP/域名/端口
- WP 内不应包含 flag
- `work/` 目录只保留必要交付物（WP + EXP；其余临时文件删除）

## Linux 操作偏好

- 如需 Linux 环境命令（例如 `file/strings/nc/socat`、容器交互、二进制分析等），优先通过 `kali_mcp` 执行；不可用时再退回本地 shell。

## 模板与辅助工具

- WP/EXP 模板见：`references/wp_template.md`、`references/exp_template.py`
- 可用脚手架生成 `work/` 产物：`scripts/scaffold_work.py`
- 可用脱敏扫描：`scripts/redaction_check.py`
